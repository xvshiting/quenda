# Quenda Project Diagnosis and Planning

> Review date: 2026-06-23

## Executive Summary

Quenda has a strong architectural direction and a clear public API story:
Kernel handles the synchronous model-tool loop, Runtime provides async session semantics, and Host is reserved for persistence and policy. The codebase already reflects that direction reasonably well.

The main management risks are not conceptual. They are operational:

- documentation and runtime constraints have recently drifted and need
  routine synchronization
- the current provider refactor is large enough to create compatibility risk
- test execution is not yet reproducible from the default shell environment
- the Host layer is still only partially realized

The project is in a good shape for continued development, but it needs a stabilization pass before adding much more surface area.

## Current Assessment

### What looks strong

- The architecture boundaries are explicit and coherent.
- Kernel is kept synchronous and deterministic, which is good for testing.
- Runtime wraps execution in async semantics without leaking lower-layer concerns upward.
- The tool system is capability-oriented instead of being a pile of ad hoc helpers.
- The project already has a decent documentation spine:
  `README.md`, `docs/getting-started.md`, `docs/api.md`, `docs/CURRENT.md`, `TODO.md`, and ADRs.

### What needs attention

- The project requires Python 3.12+, but the default shell on this
  machine resolves to Python 3.9, which breaks test collection unless
  contributors intentionally select the project interpreter.
- The provider subsystem is mid-refactor:
  - old provider modules were removed
  - new `provider.py` / `model.py` / `api/` layers were introduced
  - this is directionally right, but it needs compatibility checks and stronger tests
- `TODO.md` marks phases 0-6 complete, but some implementation and documentation details still need verification.
- The Host layer is still mostly a plan, not a product feature.

## Technical Evaluation

### Architecture

Verdict: good direction, low-to-medium risk.

The layer split is sensible and matches the project goals. The biggest architectural strength is that the Kernel is intentionally small and testable. The biggest architectural risk is over-abstracting the provider stack before the public API has stabilized.

### Code Quality

Verdict: promising, but currently in a transition state.

The code shows clear intent and fairly clean separation of concerns. However, the provider refactor suggests the codebase is at a point where regressions are likely unless compatibility is actively managed.

### Testability

Verdict: conceptually good, operationally weak right now.

The design supports deterministic testing, but the current environment mismatch prevents the test suite from being a reliable signal until the Python version and packaging path are normalized.

### Documentation

Verdict: better than average for an alpha project, but needs synchronization.

The docs are already valuable, but they need routine synchronization with active refactors. This is the kind of drift that can quietly create confusion for contributors.

## Key Risks

1. **Environment drift**
   - Risk: contributors use an interpreter below the project requirement and hit avoidable failures.
   - Impact: slow onboarding, noisy bug reports.

2. **Provider refactor regressions**
   - Risk: API shape changes break provider usage or tests.
   - Impact: public API instability during a core refactor.

3. **Host-layer deferral**
   - Risk: persistence, identity, and permissions remain vague while features accumulate above them.
   - Impact: technical debt and uncertain future integration points.

4. **Documentation drift**
   - Risk: docs describe a different reality than code.
   - Impact: contributor confusion and loss of trust in the project narrative.

5. **Missing release discipline**
   - Risk: alpha code grows without a clear stabilization milestone.
   - Impact: larger future clean-up cost.

## Recommended Priorities

### P0: Stabilize the foundation

Goal: make the project reproducible and internally consistent.

Actions:

- align all docs with the actual Python requirement
- ensure the default developer workflow uses Python 3.12+
- get the test suite green in a known-good environment
- verify that the provider refactor preserves expected public behavior

Exit criteria:

- setup docs and `pyproject.toml` agree
- tests run successfully in the intended environment
- no obvious import/path confusion in the packaged module layout

### P1: Lock the provider contract

Goal: make provider abstractions safe to extend.

Actions:

- define the stable public provider surface
- add compatibility tests around registry/model lookup
- document how old provider behavior maps to the new stack
- verify built-in providers are still discoverable and usable

Exit criteria:

- provider APIs are documented and covered by tests
- built-in providers are represented consistently
- refactor churn stops affecting the public contract

### P2: Finish the Host layer

Goal: turn the current architecture into a complete product skeleton.

Actions:

- define persistence boundaries for Session and Run
- choose the minimal storage abstraction
- clarify identity and permissions scope
- add integration tests for host-owned behavior

Exit criteria:

- persistence strategy is implemented, not just discussed
- auth/permissions responsibilities are explicit
- host responsibilities do not leak into Kernel or Runtime

### P3: Add product-facing ergonomics

Goal: make Quenda easier to adopt and operate.

Actions:

- complete the CLI story
- improve bootstrap and example workflows
- clean up docs for first-run success
- add a release checklist

Exit criteria:

- a new contributor can install, test, and run an example without guesswork
- the CLI has a clear minimal value proposition
