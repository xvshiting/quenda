# 模式系统

Quenda Code 支持三种交互模式，每种模式对应不同的行为风格和关注点。

---

## 模式概览

```
/mode chat        💬 聊天模式（默认）
/mode code        💻 代码模式
/mode architect   🏗️ 架构模式
```

---

## Chat 模式（默认）

**适用场景**: 一般问答、知识咨询、非编程任务

Chat 模式是默认模式，Agent 以通用助手的身份与你交流。

**行为特点**:
- 自由对话，无特殊约束
- 可以回答知识性问题
- 也能编写简单代码

```
> /mode chat
✅ Switched to `chat` mode.
  General conversation and Q&A
```

---

## Code 模式

**适用场景**: 编程实现、Bug 修复、代码审查、重构

Code 模式下，Agent 聚焦于编码任务，行为更加严谨。

### 行为准则

```text
💻 Code Mode

- 先思考再写代码：复杂任务先说明方案，再写代码
- 写出完整可用的代码：不留 TODO，不写伪代码
- 关注当前任务：聚焦具体需求，不做过度的泛化或重构
- 重视错误处理：关键路径加适当错误处理
```

### 代码风格要求

```text
- 遵循项目已有的代码风格和命名约定
- 优先使用标准库，必要时才加第三方依赖
- 函数/方法添加类型注解
- 公共接口添加 docstring
- 内部逻辑用注释说明"为什么"而非"是什么"
```

### 测试意识

```text
- 修改代码时，考虑对现有测试的影响
- 新功能如果有明显的测试点，提示可以怎么测试
```

### 使用示例

```
> /mode code

> 帮我实现一个 LRU Cache
# Agent 会先说明实现方案，再写出完整代码，包括类型注解和测试

> 修复这个 bug：test_main.py 报错 AssertionError
# Agent 会先读代码分析原因，再进行修复
```

---

## Architect 模式

**适用场景**: 系统设计、架构评审、技术方案讨论、迁移规划

Architect 模式下，Agent 转变为一个架构师，关注整体设计。

### 行为准则

```text
🏗️ Architect Mode

- 先全面理解问题域：确认理解了业务背景和约束条件
- 权衡利弊：每个方案讨论取舍，不只推一个方案
- 画图优于文字：使用 ASCII diagram、Mermaid 可视化架构
- 考虑演进路径：不只是"最终架构"，说明怎么从现状迁移
```

### 关注点

```text
- 模块边界和接口契约
- 数据流和状态管理
- 可测试性和可维护性
- 安全、性能、可观测性
- 与现有系统的兼容性
```

### 输出风格

```text
- 先给结论和高层概览，再逐步展开细节
- 方案建议附带实施阶段划分
- 明确指出不确定或有风险的点
```

### 使用示例

```
> /mode architect

> 我们正在建设一个电商平台，帮我设计订单系统的架构
# Agent 会先了解业务背景，然后用 Mermaid 画架构图
# 讨论不同方案的取舍，给出分阶段实施建议

> 评估一下从单体架构迁移到微服务的方案
# Agent 会分析当前架构，给出迁移路径和风险点
```

---

## 模式切换的影响

切换模式会触发：

1. **加载模式文件**：对应的 `mode-<name>.md` 指令文件被追加到系统提示词
2. **行为变化**：Agent 的回应风格和工作方式改变
3. **上下文保持**：消息历史不会丢失

模式文件在系统提示词中的位置：

```
AGENT.md (核心身份)
├── 通用指令 (principles.md, communication.md)
├── mode-code.md    ← 仅在 code 模式加载
└── mode-architect.md  ← 仅在 architect 模式加载
```

---

## 模式对照

| 对比维度 | Chat | Code | Architect |
|---------|------|------|-----------|
| 响应速度 | 快速 | 中等 | 较慢（思考更深入） |
| 输出长度 | 适中 | 较长（完整代码） | 较长（多方案） |
| 工具使用 | 按需 | 频繁（读写文件） | 较少（阅读分析） |
| 适用任务 | 问答、写作 | 编程、调试 | 设计、规划 |
| 代码质量要求 | 一般 | 高 | 不写代码 |
