# ADR-000: 语言选择与核心功能

## 状态

已接受

## 背景

Quenda 需要选择实现语言和首批核心功能，这将影响：
- 开发效率和社区采用
- 架构设计和性能特征
- 工具生态和集成能力

## 决策

### 语言：Python

**理由**：
- AI/ML 生态最成熟，与主流 LLM SDK 无缝集成
- 动态类型适合快速原型，类型注解支持渐进类型检查
- 社区庞大，易于推广和贡献

### 模型支持：可扩展 Provider 接口

**理由**：
- 避免供应商锁定
- 便于适配本地模型和私有部署
- 用户可根据需求选择后端

### 首批核心功能

1. **文件系统工具** - Agent 操作文件的基础能力
2. **会话管理** - 多轮对话和上下文管理
3. **可观测性** - 事件流、日志、调试支持

**理由**：
- 这三者是最小可用 Agent 的必备能力
- 可观测性从一开始就内置，避免后期补丁
- 为后续扩展（多 Agent 协作）打好基础

## 后果

### 正面

- 快速迭代，Python 开发效率高
- 生态系统丰富，减少重复造轮子
- 可扩展设计吸引更多用户

### 负面

- Python 性能不如编译型语言（可接受，瓶颈在模型 API）
- 动态类型需要更多测试覆盖
- 可扩展接口增加初始复杂度

### 风险

- 过度抽象可能导致接口臃肿
- 需要在灵活性和简单性之间保持平衡

## 替代方案

### 语言替代方案

| 语言 | 优点 | 缺点 |
|------|------|------|
| TypeScript | 类型安全，前后端统一 | AI 生态较小 |
| Go | 性能好，并发原生 | AI 生态有限 |
| Rust | 极致性能，内存安全 | 学习曲线陡峭，开发慢 |

### 功能替代方案

- 先做 **多 Agent 协作**：过于复杂，应先打好单 Agent 基础
- 先做 **持久化和认证**：属于 Host 层，应延后
